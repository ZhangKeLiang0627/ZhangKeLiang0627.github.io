## 写在前面

最近因为工作需求，又重新回到mcu领域好好的把玩了一回...每次温故而知新，总会发现很多以前不会注意到的细节点，比如说总线通信。

我们常常提起总线通信就会联想到串口Serial、485总线、232总线等等非常多的内容，不管是什么通信协议，大多数都是“硬件上的协议”，我这么说肯定的狭义、主观的，反正这一类的协议涉及到硬件需要连线的，在我的认知里面都是和硬件打交道的协议。

我以前学习串口的时候，看很多教程都是教你怎么初始化mcu的串口，怎么用串口发送字符串，怎么通过串口中断/接收超时来接收字符串，到这里教程就结束了。当时，我也觉得我学完了这些就代表我已经掌握了串口协议了，我会用了！

现在参加工作后再回头望，其实学会mcu怎么配置串口等等都是还在非常浅的层面，**真正的好的代码思维应该说要掌握“软件上的协议”**。可能你读到到这里都会感觉到有些懵圈哈，什么叫做软件上的协议呢，不懂（笑。

{% note info %}
**举例1：**
比方说，我们配置好了串口，可以尽情的收发字符串了。正常来说，教程普遍会告诉你，发送字符串的时候，末尾加入`\r\n`，就是告诉对方，你这包数据发完了，然后在串口中断里也是检索这个结尾来认为收到了一包完整的数据。

这确实是一个马马虎虎的朦朦胧胧的收发机制，最多测试一下串口有没有通，但是正在业务逻辑上用这样一套，那也是够呛了。
{% endnote %}

{% note info %}
**举例2：**
嗯，所以就有第二种稍微可以区分一下指令用途的方法，如AT指令，各种市面上的蓝牙模块尤其常用（下面的指令仅为演示不一定是真的AT指令哈）：
| 指令 | 功能 | 参数 |
| ---- | ---- | ---- |
| AT+ROLE? | 查询角色 | 0=从机(Slave) / 1=主机(Master) |
| AT+ROLE=0 | 设为从机 | 手机 / 电脑连接用 |
| AT+ROLE=1 | 设为主机 | 主动搜索并连接从机 |
| AT+CMODE? | 查询连接模式 | 0=指定地址 / 1=任意地址 |
| AT+CMODE=1 | 设置允许连接任意设备 | 调试常用 |

这种实现的方式也简单，先实现`\r\n`包尾判断，收到一包数据后，逐个判断前面两个字节是不是'A'、'T'，然后将整包数据和你的目标关键字匹配，比方说`strstr(uartString, "ROLE")`，然后做对应的操作。

这个只适合做那种指令简短的，信息量少的通信，而且实现比较简陋，也没有冗余校验。调试机器的时候可以用一用，比方说调试个PID，发送"P0.26"、"D1.234"，检测一下开头第一个字节是不是'P'、'I'、'D'，然后解析后面跟着的数字，`atoi()`或`atof()`字符串转对应数据类型，这样也是简单的软件上的自定义的协议...
{% endnote %}

{% note info %}
**举例3：**
还有一种是用帧头、帧尾包住对应的内容，目标机器识别对应的帧头、帧尾从而获取到包里的数据，这样就可以定义多种帧头、帧尾以接收来自不同地方的数据。

我记得我之前大学打比赛的时候，有那种上位机要显示某个数据的实时波形图的情况，我们就拿了`##data@@`这样的结构，用串口蓝牙去传输一个浮点数到上位机去做显示，非常的狼狈，但是确实挺快速有效的（笑。

当然，也有更加严密的办法，modbus、can、数据结构体打包解包（结构体末尾几个字节留给crc做校验），这种也是有的，自行下去搜看看吧...就不演示了。
{% endnote %}

## 开始

我在这里主要介绍的是一种我认为目前无论是在Web端这种高并发、还是在上下位机通信、MCU间通信都非常适合的协议，可谓是多端通吃，而且多端的协议保持一致也非常的有利于信息的转发，不用因为各种节点通信的协议不同要频繁的解包再打包，高速提效。

那么这个如此优秀的协议就是：**JSON-RPC**，[科普戳这里<-](https://zh.wikipedia.org/wiki/JSON-RPC).

我认为它好用的点在于它的可阅读性，使用最常见的JSON，然后同时包含着：发送方、接收方、指令、参数、id、crc等必备通信关键的要素，然后还要严格的遵守答复机制，收到任何符合接收要求的信息后都要返回答复。


```json
// 发送
{
  "jsonrpc": "2.0",
  "src": "device_A",        // 发送方
  "des": "device_B",        // 接收方
  "method": "your_method",  // 方法
  "params": {               // 参数
    "your_params": "xxx",
    // ...
  },
  "id": 1001,               // id
  "crc": 00,                // crc
}
```

```json
// 应答
{
  "jsonrpc": "2.0",
  "src": "device_B",        // 发送方
  "des": "device_A",        // 接收方
  "result": {               // 参数
    "sta": true,
    // ...
  },
  "id": 1001,               // id
  "crc": 00,                // crc
}
```

然后每包Json数据的末尾可以包个帧尾，比方说以检测`\r\n`为一帧结束，这样整个好用的“软件层面上的通信协议”就搭建起来啦！
